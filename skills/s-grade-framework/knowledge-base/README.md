## 核心方法论

### 验证驱动原则

**定义**：所有核心假设必须被验证，每个假设有失败标准和退出路径。

**v2.0 增强**：
- 假设依赖关系图
- 假设优先级排序（P0/P1/P2）
- 连锁反应分析

**三大子原则**：
1. 假设即需验证
2. 验证即有退出
3. 验证即有迭代

---

### 双输出设计

**定义**：每个步骤产出两种内容——主产出和情报产出。

**v2.0 增强**：
- 情报质量等级（L1/L2/L3）
- 情报存储规范

**情报质量等级**：

| 等级 | 定义 | 用途 |
|------|------|------|
| L1 | 原话记录 | 话术库 |
| L2 | 结构化数据 | 统计分析 |
| L3 | 洞察结论 | SOP 更新 |

---

### 触发器系统

**定义**：用系统代替记忆，条件触发自动化执行。

**v2.0 增强**：
- 预警触发器
- 资源触发器
- 竞品触发器

**触发器类型**：

| 类型 | 目的 | 示例 |
|------|------|------|
| 沟通 | 及时响应 | 用户联系 → 发欢迎语 |
| 交付 | 保障质量 | 数据缺失 > 30% → 告知 |
| 反馈 | 收集反馈 | 交付后 48h → 发调研 |
| 迭代 | 驱动进化 | 改进点 ≥ 3 条 → 更新 SOP |
| 预警 | 预防风险 | 进度滞后 → 原因分析 |
| 资源 | 合理分配 | 并行 > 3 → 专注提醒 |
| 竞品 | 跟踪动态 | 竞品降价 → 评估影响 |

---

### 反馈回路

**定义**：执行即迭代，每次执行都产出系统升级的原材料。

**v2.0 增强**：
- 问题分类（执行/流程/假设）
- 分类处理规则

**问题分类**：

| 类型 | 定义 | 处理方式 |
|------|------|----------|
| 执行问题 | 操作失误 | 记录，后续注意 |
| 流程问题 | SOP 缺陷 | 触发 SOP 更新 |
| 假设问题 | 前提错误 | 触发假设追踪更新 |

### 风险预案

**目的**：预设风险应对方案

**模板**：

| 风险场景 | 概率 | 影响 | 预案 |
|----------|------|------|------|
| 需求验证失败 | 中 | 高 | 换用户群/换场景 |
| 付费意愿低 | 高 | 中 | 调整定价/免费+增值 |
| 技术无法实现 | 低 | 高 | 半自动化方案 |

## 7 天执行路线图（v2.0）

```
Day 1: 需求验证 + 竞品摸底 + 资源配置
       ↓
Day 2: 半自动化 MVP + 技术验证
       ↓
Day 3-4: 种子用户 + 案例积累
       ↓
Day 5: 决策点
       ├── 反馈热烈 + 数据可标准化 → 封装
       ├── 反馈冷淡 → 重新问需求
       ├── 验证滞后 → 预警 + 原因分析
       └── 数据格式太乱 → 转纯服务
       ↓
Day 6-7: 产品封装 + 上线
```

## A级 vs S级 对比

| 维度 | A级 | S级 |
|------|-----|-----|
| **定义** | 把计划做完 | 边做边更新计划 |
| **假设** | 假设正确就继续 | 假设必须验证，有依赖关系 |
| **输出** | 单一输出 | 双输出 + 情报质量等级 |
| **触发** | 靠记忆 | 7类触发器，主动预警 |
| **复盘** | 可做可不做 | 分类复盘，精准迭代 |
| **决策** | 固定时间点 | 加速/暂停/终止机制 |
| **风险** | 出问题再处理 | 风险预案 + 预警触发 |
| **资源** | 不规划 | 资源配置模板 |
